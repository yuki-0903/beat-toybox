# BEAT RUNNER Implementation Plan

この計画書は、`BEAT RUNNER` のMVPを Next.js / TypeScript / Phaser 3 で実装するための手順をまとめたものです。

今回の前提は以下です。

- PC とスマホの両対応
- ゲーム画面はブラウザ画面いっぱいに表示
- PC は横長画面を主想定
- スマホは縦持ちプレイを主想定
- CSSでキャンバスを回転しない
- レイアウト、スケール、入力座標は Phaser 側で制御する

---

## 1. 実装方針

### 画面設計

Canvas は常に画面いっぱいに表示する。

CSS はテンプレートの方針通り、ビューポート全体を埋める役割に限定する。

Phaser 側では以下を画面サイズに応じて再計算する。

- カメラ viewport
- 背景
- 3レーンの位置
- プレイヤー位置
- 障害物の出現位置
- HUD
- タッチ入力エリア
- エフェクトの大きさ

PC/SPでゲーム性は同じにしつつ、見え方を変える。

- PC: 横長の奥行きあるランナー画面
- SP縦: 縦長画面に合わせて、走行レーンを中央に大きく配置

### レスポンシブ基準

基準解像度はテンプレートの `BASE_GAME_WIDTH` / `BASE_GAME_HEIGHT` を使う。

ただし、スマホ縦では単純な横長基準の縮小にすると遊びにくいため、以下を判定する。

```ts
const isPortrait = gameHeight > gameWidth;
```

レイアウトは `isPortrait` を使って分岐する。

- PC/横長: レーンを横方向に広めに取る
- SP縦: レーン幅を抑えつつ、プレイヤーと障害物を大きめに表示

## 3. 実装ステップ

### Step 1: ゲーム基礎レイアウト

目的: PC/SP縦どちらでも全画面で破綻しない土台を作る。

実装内容:

- `MainScene` のプレースホルダーをゲーム画面に置き換える
- 背景を全画面描画
- レーン3本を画面サイズから算出
- プレイヤーを下部中央に配置
- resize 時に全要素を再配置
- `screenScale` / `isPortrait` / レーン座標ヘルパーを作る

確認項目:

- PC横長で画面いっぱいに表示される
- SP縦サイズで画面いっぱいに表示される
- プレイヤーとレーンが画面外に出ない
- リサイズしても中央がズレない

---

### Step 2: 入力実装

目的: PC/SPで同じ操作感の移動を作る。

PC操作:

- `A` / `D`
- `←` / `→`
- `Space` でジャンプ

SP縦操作:

- 左右スワイプでレーン移動
- タップでジャンプ

実装内容:

- `InputController` を利用しつつ、ゲーム用の入力解釈を `MainScene` 側で行う
- キー長押しで連続移動しすぎないように、移動は押下トリガー単位にする
- スワイプ距離のしきい値を `balance.ts` に置く
- タップとスワイプを誤判定しないようにする

確認項目:

- PCで左右移動できる
- PCでジャンプできる
- SP縦で左右スワイプできる
- SP縦でタップジャンプできる
- タップ座標と実際の反応がズレない

---

### Step 3: ランナー挙動

目的: 「走っている」気持ちよさの最低限を作る。

実装内容:

- プレイヤーのレーン移動アニメーション
- ジャンプアニメーション
- 地面またはレーンの流れ演出
- 簡易スピード感の背景ライン
- プレイヤーの誇張リアクション

確認項目:

- レーン移動が瞬間移動ではなく気持ちよく見える
- ジャンプ中の判定状態がわかる
- SP縦でもプレイヤーが小さくなりすぎない

---

### Step 4: BPM同期システム

目的: 障害物や演出を拍に合わせて動かす。

実装内容:

- BPM設定を `balance.ts` に追加
- `beatIntervalMs` を算出
- 経過時間から現在拍を計算
- 拍ごとにイベントを発火
- 小節単位のパターン管理を作る

最初はBGM実ファイルなしでも動くようにする。

確認項目:

- 一定BPMで拍が進む
- FPSが揺れても拍タイミングが大きくズレない
- 一時停止やリスタートでタイマーが破綻しない

---

### Step 5: 障害物生成

目的: リズムに合わせて避ける対象を出す。

実装内容:

- 障害物の種類を作る
  - レーン障害物
  - ジャンプ障害物
- 拍または小節パターンに合わせて生成
- 奥から手前に流れてくるように移動
- 判定ラインをプレイヤー付近に設定
- 画面外の障害物を破棄

確認項目:

- 3レーンのどこかに障害物が出る
- BPMに合わせて出現する
- PC/SP縦どちらでも判定ラインが見やすい
- 障害物が画面外に残り続けない

---

### Step 6: 回避・MISS判定

目的: 避けるゲームとして成立させる。

実装内容:

- 判定ライン通過時に成功/失敗を判定
- 同じレーンで地上障害物に当たるとMISS
- ジャンプ障害物はジャンプ中なら成功
- 成功でコンボ加算
- MISSでコンボリセット
- MISS時に画面/音/プレイヤーリアクションを入れる

確認項目:

- 避けた時だけコンボが増える
- 当たった時にコンボが切れる
- 判定が理不尽に感じない
- SP縦でも障害物の種類が見分けられる

---

### Step 7: 音フィードバック

目的: 操作と成功が音楽の一部になる感覚を作る。

実装内容:

- ブラウザ音制約に合わせて、START操作後に音を有効化
- 操作SEを追加
  - 左移動
  - 右移動
  - ジャンプ
  - 回避成功
  - MISS
- BGMなしでもSEだけでリズム感が出る状態にする
- 後でBGMを追加できるようにキー名を安定させる

確認項目:

- 初回操作前に音を鳴らそうとしない
- PC/SPで音が鳴る
- SEが重なりすぎて不快にならない
- MISS音が連続再生されすぎない

---

### Step 8: コンボ演出

目的: うまく避け続けるほど世界が気持ちよく変化する状態を作る。

実装内容:

- HUDにコンボ表示
- コンボ段階で背景色やレーン演出を変化
- 成功時の小エフェクト
- 一定コンボごとに音/見た目を強化

確認項目:

- コンボが視認しやすい
- HUDがPC/SPで邪魔にならない
- 演出がゲーム視認性を壊さない

---

### Step 9: FEVER

目的: 連続成功時の盛り上がりを作る。

実装内容:

- FEVER突入コンボ数を `balance.ts` に定義
- FEVER中の速度補正
- 背景/粒子/カメラ演出を強化
- FEVERゲージまたは表示を追加
- MISSでFEVER解除または弱体化

確認項目:

- FEVER突入が気持ちよい
- 速度UPしても理不尽にならない
- SP縦で演出が過密にならない

---

### Step 10: スタート・リザルト

目的: 1プレイの流れを完成させる。

実装内容:

- START画面
- プレイ開始
- 一定時間または一定小節でGOAL
- リザルト表示
  - スコア
  - 最大コンボ
  - MISS数
  - ランク
- リトライ

確認項目:

- START操作で音が解放される
- GOAL後に結果が見える
- リトライで状態が初期化される
- PC/SP縦でUIがはみ出さない

## 6. PC/SP縦レイアウト方針

### PC

- レーンは画面中央に広めに配置
- HUDは上部左右に配置
- プレイヤーは画面下から少し上
- 障害物は奥から手前に流れる見た目

### SP縦

- レーンは画面中央に縦長に配置
- HUDは上部にコンパクト配置
- プレイヤーは下部だが、指で隠れすぎない高さに置く
- タップ/スワイプ入力は画面全体で受ける
- 重要な文字は小さくしすぎない
